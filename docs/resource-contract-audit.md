## Resource Contract Default

ADR 0005 says new durable, agent-facing, provider-facing, CLI-facing, cross-product, or user-authored Kontour contracts should default to a Kubernetes-inspired Resource Contract shape:

- `apiVersion` and `kind` identify the product namespace and resource type.
- `metadata` carries identity, labels, annotations, ownership, timestamps, source references, and integrity anchors.
- `spec` carries desired intent: the standard, requirement, authority, check plan, target, or configuration the user or system wants applied.
- `status` carries observed facts: readiness result, evidence coverage, generated paths, current authority state, or projection result.
- `status.conditions[]` summarizes important observed states with type, status, reason, message, severity, observed generation, and timestamps where useful.

The intent/status split matters for Veritas. Repo Standards, Repo Maps, authority settings, and claim definitions mostly express desired governance intent. Readiness reports, evidence records, conformance summaries, standards feedback, Surface projections, and runtime observations mostly express observed status. Evidence remains separately inspectable; a Resource Contract wrapper should link evidence rather than hide it in metadata.

`CONTEXT.md` currently describes a Kontour Resource Shape with `apiVersion`, `kind`, `metadata`, `spec`, optional `status`, and optional `proof`. This audit treats `proof` as a Veritas evidence/provenance mapping candidate, not an alternate required shape. Future Veritas migrations should map `proof` content into `metadata`, `status`, `status.conditions`, and linked Evidence, Attestation, Approval, or Surface trust artifacts unless a narrow contract needs a dedicated provenance field.

## Classification

Audience and durability:

- `durable/user-authored`: reviewed repo-local source of governance intent.
- `durable/generated`: generated evidence or summaries that humans, agents, CI, or providers inspect.
- `provider-facing`: records that bind Veritas to external authority, approval, runtime, or Surface systems.
- `CLI-facing`: JSON printed by public commands for automation.
- `internal`: implementation-only helper shapes that should not be treated as stable contracts.

Recommendations:

- `migrate`: adopt Resource Contract shape in a future slice.
- `map`: the current external contract is documented and tested; an explicit Resource Contract mapping is backlogged.
- `exception`: keep product-native shape because wrapping it would make this specific contract less clear.
- `defer/observe`: revisit after adjacent product work or dependency alignment.
- `internal/not-a-contract`: do not publish as a Resource Contract.

## Exception Rationales

- Repo conformance Markdown summaries: exception because they are human presentation derived from JSON. Resource shape would make them less clear by mixing readable prose with machine lifecycle fields and duplicating the canonical JSON report.
- Marker benchmark raw session logs and scenario examples: exception because their useful structure is scenario/turn/comparison data. Resource shape would add repetitive wrappers around examples and turns, making benchmark diffs harder to review.
- Raw Agent Session Logs: exception because the raw shape is owned by Claude Code, Codex, or another runtime. Resource shape would obscure provider-native fields and increase privacy and retention ambiguity; normalized observations should be mapped instead.
- Internal implementation helper shapes: exception because they are transient local data. Resource identity, metadata, and conditions would imply durability and compatibility that the code does not promise.

## Migration Slices

Every `migrate` recommendation above needs a separate implementation slice. Suggested order:

1. Repo Standards Resource slice
   - Add `RepoStandards` Resource Contract schema or compatibility wrapper while preserving current `rules[]`.
   - Verify with schema fixture validation for [`repo-standards/work-agent-convergence.repo-standards.json`](../repo-standards/work-agent-convergence.repo-standards.json), `npm run verify`, and a readiness run against a repo using the migrated standards.

2. Repo Map Resource slice
   - Add `RepoMap` and graph/resource mapping for work areas, boundaries, protected areas, and evidence routing.
   - Verify with schema validation for [`repo-maps/work-agent.repo-map.json`](../repo-maps/work-agent.repo-map.json), boundary check fixtures, and `npm run verify`.

3. Readiness Report and Evidence Resource slice
   - Introduce `ReadinessReport` or `EvidenceRecord` Resource Contract output with desired source/check plan in `spec` and readiness/evidence/Surface projection in `status`.
   - Verify with existing evidence examples in [`examples/evidence/`](../examples/evidence/), CLI JSON snapshots for `veritas readiness --check evidence --format json`, Surface projection compatibility, and `npm run verify`.

4. Attestation Resource slice
   - Migrate attestation records to a Resource Contract or add a versioned compatibility wrapper with approval resolution in `status`.
   - Verify with bootstrap and policy-change attestation tests, hash drift checks, approval resolver fixtures, and `npm run verify`.

5. Authority Settings Resource slice
   - Move verification authority policy into `spec` and add validation/status conditions for resolver reachability and approval policy mode.
   - Verify with authority settings schema fixtures, `attest status`, approval resolver tests, and `npm run verify`.

6. Standards Feedback Resource slice
   - Migrate draft and confirmed feedback records with `spec.evidenceRef`, `spec.mode`, `status.outcome`, `status.measurements`, `status.governance`, and `NeedsConfirmation` conditions.
   - Verify with [`examples/standards-feedback/`](../examples/standards-feedback/), session-log observation tests, history JSONL behavior, and `npm run verify`.

7. Standards Recommendation Resource slice
   - Represent proposed standards changes as `StandardsRecommendation` resources with proposed intent in `spec` and lifecycle/apply result in `status`.
   - Verify with recommendation generate/list/show/apply CLI tests, diff payload fixtures, and `npm run verify`.

8. Repo Conformance Resource slice
   - Migrate `.kontourai/veritas/repo-conformance/*.json` summaries to `RepoConformanceReport` resources while leaving Markdown summaries as presentation.
   - Verify with [`examples/repo-conformance/`](../examples/repo-conformance/), `npm run veritas:conformance:verify`, `npm run verify`, and a check that `latest.json` points to the migrated report.

9. CLI JSON Resource summary slice
   - Normalize CLI JSON that returns created artifact paths or statuses so automation can find `apiVersion`, `kind`, `metadata.name`, and resource references when durable artifacts are involved.
   - Verify with CLI tests for readiness, init, attest status, integrations status, feedback, recommendations, claims, and `npm run verify`.

10. Guided Init Plan Resource slice
   - Define an `InitRecommendation` or `SetupPlan` resource wrapper for `.veritas/init-plans/*.json`, keeping existing `artifact_payloads` and `artifact_hashes` compatibility while mapping target root, recommended artifacts, owner answers, and apply lifecycle into `spec`/`status`.
   - Verify with `init --explore --output`, `init --guided --answers --output`, `init --apply --plan`, hash mismatch rejection, output-directory constraints, and `npm run verify`.

## Mapping And Deferred Slices

- Evidence Inventory: define an `EvidenceInventory` mapping after freshness and item ownership are clarified. Verify with coverage output and inventory manifests.
- Approval Records: keep provider-native approval references and map resolver observations into attestation/resource status. Verify with offline approval resolver cases.
- Marker Benchmark suite reports and scores: map suite-level outputs to Resource status after benchmark reporting stabilizes. Verify with marker benchmark examples and score comparison tests.
- Claim Inputs, Claim Store, and Surface Read Models: defer schema migration until Surface Resource Contract alignment is defined. Verify with Surface TrustBundle validation and `docs/veritas-and-surface.md` boundary examples.
- Runtime Integration installed config/status: map generated integration config to resource summaries without changing runtime hook protocols. Verify with `integrations <tool> status` CLI tests and generated hook fixture checks.
- Guided init plans: if not migrated with the CLI JSON slice, map `.veritas/init-plans/*.json` as setup-plan resources in a separate migration so reviewed bootstrap intent, payload hashes, owner answers, and apply status are inspectable by agents.

## Open Questions And Risks

- Surface ownership: Veritas should not migrate Surface-owned claim or trust report schemas ahead of Surface's own Resource Contract decision.
- Backward compatibility: existing `.veritas/` artifacts and examples use current field names such as `version`, `schemaVersion`, and `record_schema_version`; migration needs compatibility readers or explicit breaking-version guidance.
- Proof terminology: future work should decide whether Veritas keeps a visible `proof` field for user-facing provenance or maps that language entirely into `metadata`, `status`, `conditions`, and linked evidence.
- CLI compatibility: stdout JSON may be consumed by automation. Migration should version or wrap outputs without silently removing current keys like `artifactPath`, `reportArtifactPath`, `draftArtifactPath`, and `standardsFeedbackArtifactPath`.
- Privacy and retention: Agent Session Logs and raw runtime inputs should stay provider-native inputs unless a narrow, redacted observation resource is defined.
